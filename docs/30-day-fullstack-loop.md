# 30 天 AI 辅助全栈闭环

## 目标与时间盒

- 开始日期：2026-07-19
- 目标日期：2026-08-18
- 建议投入：每周 5–6 天，每次约 2–3 小时；若实际时间变化，优先调整范围而不是把计划写成已完成。
- 项目载体：`apps/mini-saas/`

第一轮目标不是成为资深后端，而是能够借助 AI 从需求走到线上，并具备指导、审查和纠正常规后端实现的基础判断力。

## 闭环完成标准

在目标日期前完成并验证：

1. 前端能够注册、登录并保持认证状态。
2. 已登录用户能够创建、查询、更新和删除自己的项目。
3. 项目保存在 PostgreSQL，应用重启后数据仍然存在。
4. 未登录、越权、输入错误和资源不存在具有正确响应。
5. 数据库变更通过迁移管理，关键约束由数据库保护。
6. 核心 Service 规则有单元测试，关键用户流程有 e2e。
7. API 与 PostgreSQL 可以通过 Docker 在本地启动。
8. 前端、API 和数据库至少完成一次可访问的部署。
9. 学习者能从浏览器请求追踪到 Controller、Service、数据访问和数据库，并能根据日志定位常见故障。

## 四周里程碑

### 2026-07-19 至 2026-07-25｜API、CRUD 与数据库

- 完成 Projects 进程内 CRUD。
- 学习 PostgreSQL、表、行、主键和基本约束。
- 选择 ORM，建立数据库连接和第一份迁移。
- 将 Projects CRUD 改为真实持久化。

### 2026-07-26 至 2026-08-01｜用户、认证和权限

- 建立 User 与 Project 的归属关系。
- 实现注册、登录和密码哈希。
- 确定认证凭证方案并实现 Guard。
- 验证用户不能读取或修改其他用户的项目。

### 2026-08-02 至 2026-08-08｜联调、测试与本地运行

- 前端接入认证和项目 API。
- 补齐错误处理、环境配置和 OpenAPI。
- 覆盖关键单元测试与 e2e。
- 使用 Docker Compose 运行 API 与 PostgreSQL。

### 2026-08-09 至 2026-08-18｜部署与验收

- 部署前端、API 和数据库。
- 执行生产迁移并配置 CORS、Cookie、HTTPS 和密钥。
- 完整验证注册、登录、权限和项目 CRUD。
- 使用日志完成一次真实故障定位。
- 依据闭环标准复盘缺口，形成第二轮深化清单。

## 第一轮不可跳过

- 外部输入运行时校验。
- 数据库迁移、主键、唯一约束和项目归属。
- 密码哈希、认证和服务端权限校验。
- 密钥与环境变量安全。
- 关键业务规则测试和完整 HTTP e2e。
- 部署后的真实访问与日志验证。

## 第一轮暂缓深化

- 微服务和复杂分布式架构。
- Redis 深入原理、队列和分布式锁。
- 高并发、复杂事务和系统性能调优。
- Kubernetes 和复杂 CI/CD。
- 完整可观测性平台与高级安全攻防。

这些内容不是不学习，而是在第一轮闭环完成后进入第二轮，避免尚未走通端到端流程时过早扩展。

## 每次课程的默认节奏

1. 恢复进度并明确本次唯一主目标。
2. 解释该知识在完整请求链路中的位置。
3. 由 Agent 完成主体实现，学习者先预测行为并审查关键 diff。
4. 只安排能增强理解的短小手写练习。
5. 通过测试、真实请求或运行证据验证结果。
6. 更新学习档案，提交并推送完整阶段。
